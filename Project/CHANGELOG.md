## v3.1

Magic-number and persistence simplification. MT5 magic numbers are treated as user-assigned EA identifiers on trade requests/orders/deals/positions, not a platform-enforced ownership lock. v1 duplicate-magic detection is therefore terminal-local only, using a simple Global Variable marker for `(account, symbol, InpMagicNumber)`; no VPS/account-wide lease is introduced. v1 persistence is GV-only plus magic-filtered broker history; the custom file snapshot layer and per-write `FileFlush` requirement are deferred out of v1 to avoid unnecessary I/O. Ticket IDs are stored losslessly in GV as high/low numeric chunks. One strategy attaches to one symbol; one symbol may host multiple strategies with different magics.

## v2.6

## v2.5

Rejected the exclusive-symbol GV lease model. Netting accounts now support multiple TradeSpine EAs on the same symbol through a per-EA virtual ownership ledger plus a short-lived execution mutex. Updated close/SL/trailing semantics for netting, changed lot-cap defaults to skip rather than silently clamp, fixed stale root/include/session wording, and added acceptance criteria for quoted includes and same-symbol multi-EA netting behavior. **Core design principle locked: netting accounts MUST support multiple EAs per symbol — the per-symbol isolation is the adapter's responsibility, not an operating constraint.** The only structural difference between hedging and netting at the framework level is the close/SL/TP mechanism (broker-native tickets on hedging; framework-owned pending stop/limit orders linked to a virtual ledger on netting). Strategy code is mode-agnostic.

## v2.1

Added statistical-validation division to the analytics model: PSR (Probabilistic Sharpe Ratio) is the in-loop per-pass optimization criterion (single-strategy; computable in `OnTester()` since it needs only the pass's own returns/skew/kurtosis/length); ENUM_OPT_CRITERION (v3 selectable in-loop optimization fitness function — balance-DD / balance-stability / PSR / profit-factor / expected-payoff; fixed per run, never swept); DSR (Deflated Sharpe Ratio) is the post-optimization audit run in the Python ETL (multi-trial; needs trial count + variance-of-Sharpes-across-passes + clustering for effective independent N, none of which a single `OnTester()` call can see). Documented the optimization-output data requirements needed for Python to compute DSR (per-pass Sharpe + parameter vector + pass count via `OnTesterPass`/`FrameAdd`). v2/v3 roadmap updated. No v1 scope change — this is analytics-layer guidance.
