Technical Architecture · PRD v3.75

TradeSpine
Architecture

A modular, reproducible foundation for B3 futures strategies on MetaTrader 5. One strategy attaches to one chart symbol, one symbol may host multiple strategies, and all trade flow passes through guarded, strategy-scoped infrastructure.

02

Object Composition & Ownership

Solid arrows are owns (a member instance); dashed arrows are uses-via-interface. The coordinator depends on narrow interfaces so tests can vary sizing, stops, position state, and submission independently. Netting state is strategy-scoped through the virtual gross-position ledger; duplicate magic and execution locks are terminal-local Global Variables. The inner CTrade is private inside CGuardedTrade — strategies cannot reach it.

03

Intent → Execution Pipeline

All entries flow through one imperative path: strategies call OpenLong/OpenShort from OnNewBar or OnTickEvent; close helpers in OnManagePosition route through the coordinator close branch. v1 executes only on RETAIL_HEDGING, where close means this EA's magic-filtered tickets. Netting and exchange-netting selections fail initialization as deferred v2+ modes. The audit record is strategy-scoped and paired before/after execution.

04

Position State Machine

This state machine is per strategy instance lifecycle. In v1 hedging mode, strategy ownership maps to this EA's magic-filtered broker tickets and GV-backed state. Netting aggregate-position and virtual-ledger lifecycles are deferred to v2+. PENDING_ENTRY exits through success, broker rejection, or framework timeout; cancel-vs-reject remains distinct to avoid phantom-position bugs.

stateDiagram-v2
  [*] --> FLAT
  FLAT --> PENDING_ENTRY: entry submitted
  PENDING_ENTRY --> ACTIVE: DEAL_ADD (in) confirmed
  PENDING_ENTRY --> FLAT: broker REJECT/INVALID
(origin a, immediate)
  PENDING_ENTRY --> PENDING_CANCEL: framework timeout
cancel requested
  PENDING_CANCEL --> FLAT: cancel confirmed
  PENDING_CANCEL --> HALT: cancel failed or
no confirmation
  ACTIVE --> PENDING_EXIT: close submitted
  PENDING_EXIT --> FLAT: DEAL_ADD (out) confirmed
  ACTIVE --> FLAT: external close
(reason = External, WARN)
  ACTIVE --> HALT: emergency SL failure
(F-POS-19)
  HALT --> FLAT: reconcile-safe auto-clear
(F-POS-20; on restart)
  HALT --> [*]: EA removed / OnDeinit

  note right of PENDING_ENTRY
    rejection path: ownership flag false
    timeout path: InpFillTimeoutSecs (5s)
    state machine tolerates skipped/
    out-of-order tester events
  end note
  note right of PENDING_CANCEL
    framework-initiated cancel

    distinguish from broker reject
  end note

06

Lifecycle Staging

Four guaranteed phases, each logged at INFO. Trading is allowed only in phase 4 — a trade call from a constructor or from OnStrategyInit fails fast with an error naming the stage, rather than firing an order before the guards are wired. This is the discipline borrowed from NautilusTrader's "don't touch services before registration" rule.

07

Catastrophic Defense in Depth

The 1066-lot scenario must be impossible. Three independent layers each get a veto, and the order only reaches the vendored CTrade after passing every guard stage. The seven sequential stages add a broker-level OrderCheck pre-flight (cumulative margin + trading-disabled detection) BEFORE our own margin check, then a retry loop on the inner execute stage handles REQUOTE/PRICE_CHANGED/connection-class retcodes with bounded retries — distinct from both rejections (re-arm) and timeouts (HALT).
